# ใช้ agent-teams แล้วเปลือง quota Claude น้อยลง — โดยที่คุณไม่ต้องจัดการเอง

> สิ่งที่ผู้ใช้หลายคนสังเกตเห็นจริง: ทำงาน research / coding / เขียน docs ผ่าน agent-teams
> แล้ว **% โควต้าต่อ 5 ชั่วโมง / ต่อสัปดาห์ ของ subscription Claude ลดลงชัดเจน**
> ที่ดีคือมันเกิดจาก **ดีไซน์ของระบบ ไม่ใช่สิ่งที่คุณต้องมานั่งบริหาร token เอง**

---

## TL;DR

agent-teams ลดภาระ (burden) เรื่อง quota ให้คุณ 3 ทางหลัก — **ทั้งหมดทำงานเอง:**
1. เทงานอ่านหนัก ๆ ไปไว้ใน subagent ที่ใช้แล้วทิ้ง → เซสชันหลักไม่อืด
2. system prompt นิ่ง → ได้ประโยชน์จาก caching อัตโนมัติ
3. context ถูกแยกตาม project/role → ไม่ลากของไม่เกี่ยวมาเปลือง quota

คุณแค่ **บอกเป้าหมาย** ที่เหลือระบบจัดการให้ — ไม่ต้องนั่งนับ token, ไม่ต้องคอยล้าง context เอง

---

## เทียบชัด ๆ: Claude Code ปกติ vs ผ่าน agent-teams

> agent-teams **รันอยู่บน Claude Code** อยู่แล้ว — Claude Code ปกติ "ทำแบบนี้ได้เหมือนกัน"
> ถ้าคุณมีวินัยจัดการเอง สิ่งที่ agent-teams เพิ่มคือ **ทำให้แพตเทิร์นประหยัดเป็นค่าเริ่มต้น
> อัตโนมัติ + reuse ได้ + มี tracking** = ภาระตกที่ระบบ ไม่ใช่ที่คุณ

| ด้าน | Claude Code ปกติ | ผ่าน agent-teams |
|---|---|---|
| เทงานหนักให้ subagent | ทำได้ แต่ต้องสั่ง + เขียน prompt เองทุกครั้ง | Lead แตกงาน + delegate ให้เอง โดยดีฟอลต์ |
| context เซสชันหลัก | โตเรื่อย ๆ ถ้าคุยยาวไม่ระวัง | แบน — ของหนักไปอยู่ subagent อัตโนมัติ |
| reuse บริบท/วิธีทำงาน | เริ่มใหม่/อธิบายซ้ำทุก session | playbook + Kanban + shared เก็บไว้ reuse |
| caching | ได้ถ้า prefix นิ่ง (แชตยาวมัก ทำ prefix ขยับ) | system prompt โครงสร้างนิ่ง = cache เสถียรกว่า |
| scope ของ context | คุณจัดเอง (เสี่ยงของไม่เกี่ยวปน) | แยกตาม project/role ให้อัตโนมัติ |
| เห็น cost/quota | ดูรวมจากตัว usage | cost tracker ต่อ session + budget soft-warn |
| **ภาระผู้ใช้** | ต้องมีวินัยจัดการ context/token เอง | ระบบจัดให้ — โฟกัสที่งาน |
| **ผลต่อ quota (งานยาว+อ่านเยอะ)** | **ขึ้นกับวินัยผู้ใช้** | **ลดลงโดยดีฟอลต์** |

**ตัวเลขที่วัดได้จริง:** prompt-caching ของ specialist context ที่นิ่ง วัดได้ **input-cost ลดลง ~77.5%**
ในสถานการณ์ 10 รอบ (ดู README) — ไม่ใช่แค่ทฤษฎี

## ระบบช่วยอะไรให้อัตโนมัติบ้าง (= ลด burden คุณ)

### 1. Lead เทงานหนักให้ผู้เชี่ยวชาญเอง
คุณบอกแค่ "ช่วย research/แก้โค้ด/เขียนคู่มือเรื่องนี้" → Lead แตกงานแล้ว spawn specialist เอง
การ **อ่านไฟล์/ค้นข้อมูล/อ่าน log ก้อนใหญ่** ไปเกิดในบริบทของ subagent
**คุณไม่ต้องตัดสินใจเองว่าอันไหนควร offload** — มันเป็นพฤติกรรมมาตรฐานของระบบ

### 2. Context แยก + ทิ้งให้อัตโนมัติ
subagent ทำงานในหน้าต่าง context ของตัวเอง เสร็จแล้ว**คืนกลับมาแค่บทสรุป**
ของหนักที่มันอ่านไม่ถูกลากกลับเข้าเซสชันหลัก → **main thread ผอมอยู่ตลอด**
คุณ **ไม่ต้องคอยเคลียร์ context หรือกังวลว่าแชตจะอืด** ระบบทิ้งบริบทชั่วคราวให้เอง

### 3. Caching มากับโครงสร้าง
CLAUDE.md / team playbook / นิยาม agent เป็น prompt ที่ **นิ่ง** → เข้าเกณฑ์ cache
ทำงานซ้ำ ๆ บน project เดิม = ได้ **cache-read** (นับเข้า quota น้อยกว่า input สด) อัตโนมัติ
**ไม่ต้อง config อะไร**

### 4. Context ถูก scope ตาม project / role
แต่ละ project มี context ของตัวเอง (shared / role-state) งานคนละโปรเจกต์ไม่ปนกัน
→ ไม่มี "ของไม่เกี่ยว" ถูกแบกไปเปลือง quota โดยใช่เหตุ

### 5. มองเห็น cost โดยไม่ต้องนั่งนับเอง
ระบบจับ token / cost ต่อ session ให้ (cost tracker) และ **soft-warn เมื่อใกล้งบ**
→ คุณได้ภาพการใช้งานโดยไม่ต้องทำบัญชี token เอง

## ทำไมมันลด quota ได้จริง (กลไกสั้น ๆ)

| | raw chat ธรรมดา | ผ่าน agent-teams |
|---|---|---|
| ของหนัก (ไฟล์/log/source) | อยู่ใน main thread ถูกส่งซ้ำ **ทุกเทิร์น** | อ่าน **ครั้งเดียว** ใน subagent แล้วทิ้ง |
| ต้นทุนสะสมเมื่อคุยยาว | โตแบบ **quadratic** (ยิ่งยาวยิ่งพุ่ง) | โตแบบ **linear** (เซสชันหลักแบน) |
| prompt ซ้ำ | จ่ายเต็มทุกครั้ง | ได้ **cache-read** |

ผลรวม: **เซสชันยาว ๆ กิน quota ช้าลงเห็นได้ชัด**

## เห็นผลชัดกับงานแบบไหน (ตั้งความคาดหวังให้ถูก)

- ✅ **ได้เต็ม:** research, coding, เขียน docs/คู่มือ — งาน **ยาว + อ่านเยอะ + delegate ได้**
- ➖ **ได้น้อย:** ถาม-ตอบสั้น ๆ จบในเทิร์นเดียว (งานเล็กเกินกว่าจะคุ้มค่า overhead การ spawn)

---

## นิสัยเล็ก ๆ ที่ช่วยเพิ่ม (ทำง่าย ไม่ต้องลงแรงมาก)

1. **บอกเป้าหมายแล้วปล่อยให้ Lead delegate** — อย่าลากงานอ่านหนัก ๆ มาทำเองใน main thread
2. **งานคนละเรื่อง = เปิด session ใหม่** อย่าให้ context ก้อนเก่าค้างลากยาว
3. **ทำงานบน project เดิมซ้ำ ๆ** เพื่อให้ได้ cache hit

> ที่เหลือระบบจัดการให้ — เป้าหมายคือคุณโฟกัสที่ "งาน" ไม่ใช่ "การประหยัด token"

---

## หมายเหตุเชิงเทคนิค (เพื่อความตรงไปตรงมา)

- ที่ลดลงคือ **% โควต้า subscription** (จาก offload + caching) — เป็นมิเตอร์จริงของ Claude เชื่อถือได้
- "จำนวน token รวมทั้งระบบ" ไม่จำเป็นต้องน้อยลงเสมอ — งานยังมีต้นทุน เพียงแต่ถูกจัดวางให้คุ้มกว่า
  ดังนั้นเวลาประเมิน ให้ดู **% quota / cost ($)** ไม่ใช่นับ token ดิบ ๆ
- อย่าใช้ตัวเลข token ที่ตัว agent **พิมพ์รายงานเอง** เป็นหลักฐาน (โมเดลประเมิน cost ตัวเองไม่แม่น) —
  ดูจากตัวบอก usage ของ Claude / cost tracker เท่านั้น
