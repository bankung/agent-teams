# คู่มือใช้งาน: netops-monitoring-reader (ผู้ช่วยวิเคราะห์ alert จาก Zabbix)

> เวอร์ชัน PoC — read-only diagnosis เท่านั้น ใช้ทดลองแล้วส่ง feedback กลับมาปรับได้

## 1. มันช่วยอะไร / ทำอะไรไม่ได้

**ช่วย:** อ่านข้อมูล problems / events / history จาก Zabbix → จับกลุ่ม alert ที่เกี่ยวเนื่องกัน (correlate) → บอกว่า **root cause ตัวจริงน่าจะคืออะไร** + แนะนำว่าควรไปเช็ค/แก้อะไร (เป็นคำสั่งของอุปกรณ์ vendor นั้น ๆ ให้เอาไปใช้เอง)

**ทำไม่ได้ (สำคัญ — ปลอดภัยไว้ก่อน):**
- ❌ ไม่แตะ config อุปกรณ์ ไม่สั่งอะไรลง MikroTik / Fortinet / Ubiquiti
- ❌ ไม่ acknowledge / close problem ใน Zabbix
- ✅ มัน **แนะนำอย่างเดียว — คนเป็นคนตัดสินใจและลงมือแก้เอง**

เหมาะกับตอน **alert เด้งพรึบพร้อมกันหลายตัวแล้วงงว่าตัวไหนคือตัวต้นเหตุ**

## 2. เตรียมข้อมูล (เลือกแบบใดแบบหนึ่ง)

**แบบ A — ง่ายสุด เริ่มที่นี่ (ไม่ต้องตั้งค่าอะไร):**
1. เข้า Zabbix → **Monitoring → Problems**
2. ปรับช่วงเวลา/severity ตามที่อยากดู
3. กด **Export to CSV** (มุมขวาล่างของตาราง) หรือก๊อปทั้งตารางมาก็ได้
4. เอาไฟล์/ข้อความนั้นมาวางในขั้นตอนที่ 3

> ขอให้มีคอลัมน์พวกนี้ติดมาด้วยถ้าได้: Time, Severity, Host, Problem, Duration, Tags

**แบบ B — สำหรับใช้ประจำ/อัตโนมัติ (ทีหลัง):** ขอ admin สร้าง Zabbix user แบบ **read-only** + API token แล้วแจ้งทีม — ระบบจะดึงเองได้ ไม่ต้อง export มือ (จำเป็นถ้าจะทำ "ตรวจอัตโนมัติทุกชั่วโมง")

## 3. วิธีเรียกใช้

ในเซสชันที่ผูกกับโปรเจกต์ **netops** พิมพ์คำขอ + แนบข้อมูล Zabbix ที่เตรียมไว้ เช่น:

```
ช่วยวิเคราะห์ Zabbix problems ชุดนี้หน่อย ว่าตัวต้นเหตุจริง ๆ คืออะไร
ควรไปเช็ค/แก้ตรงไหนก่อน:

<วางข้อมูล Problems ที่ export มาตรงนี้>
```

ตัวอย่างคำขอแบบอื่น:
- เจาะจง host: `host CORE-SW-01 หลุดบ่อย ช่วยดู history + อุปกรณ์ที่ต่อพ่วงให้หน่อย: <paste>`
- ถามภาพรวม: `คืนนี้ alert เด้ง 25 ตัว จัดกลุ่มให้หน่อยว่ามีกี่ปัญหาจริง ๆ: <paste>`

## 4. ผลลัพธ์ที่จะได้ (โครงสร้าง)

1. **สรุปอาการ** — มี alert กี่ตัว เริ่มเด้งตั้งแต่เมื่อไหร่
2. **การจับกลุ่ม + root cause ที่น่าจะเป็น** — เช่น "18 จาก 25 ตัวเป็นผลพวงจาก uplink ของ CORE-MIKROTIK-01 ตัวเดียว down"
3. **หลักฐาน** — อ้าง trigger / item / เวลา ที่ใช้สรุป
4. **คำสั่งที่ควรเช็คเอง** (แยกตาม vendor):
   - MikroTik: `/interface print`, `/ip route print`, `/log print`
   - Fortinet: `get system interface`, `diagnose ...`, ค้น log
   - Ubiquiti: ดู AP/device health, uplink ใน controller
5. **ระดับความมั่นใจ + วิธียืนยัน** — "ถ้า ... จริง = ใช่; ถ้า ... = ไม่ใช่"
6. **แจ้งจุดบอด** — ถ้า host/กลุ่มไหนไม่มีข้อมูล จะบอก ไม่เงียบแล้วเหมาทันทีว่าปกติ

## 5. เคล็ดลับให้ผลแม่นขึ้น

- แนบ **ช่วงเวลาที่ alert เริ่มเด้ง** (ช่วยจับ storm onset)
- บอก **context ล่าสุด** เช่น เพิ่งเปลี่ยน config / มี maintenance / ไฟดับ → ช่วยให้ตัดสาเหตุได้เร็ว
- ถ้ามีผัง topology / host dependency แนบมาได้ยิ่งดี (ช่วย correlate แม่นขึ้น)

## 6. ข้อควรระวัง

- เป็น **read-only ปลอดภัยต่อ network** แน่นอน — แต่ **คำแนะนำการแก้ต้องอ่าน/ตรวจก่อนลงมือเสมอ** อย่าก๊อปคำสั่งยัดลงอุปกรณ์โดยไม่เข้าใจ
- ก่อนวางข้อมูล ลบของลับที่ไม่จำเป็น (password ที่หลุดมาใน log, public IP ที่อ่อนไหว) ออกก่อน
