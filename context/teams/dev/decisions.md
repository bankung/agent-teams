# Architectural & process decisions — dev team

> **Lead is the only writer of this file.** Subagents propose updates in their final report — Lead reviews, may ask the user, then writes the entry.
>
> **Scope:** decisions about the **dev team methodology / orchestration system itself** — Tier-1 smoke flow, Tier-2 release wrap-up, lifecycle steps, agent prompts, Bucket architecture. Applies to **every** `team='dev'` project, not to any one project's data model or endpoints.
>
> Per-project decisions (a project's data model, endpoints, UI, migrations, deps) live in that project's `context/projects/<project>/shared/decisions.md`. When a project-specific incident produces a methodology lesson, the incident decision goes in the project file and the methodology decision goes here, cross-linked.
>
> Format: append-only log. Newest entry at the top. Each entry has a date, scope, who proposed it, and the reasoning.
>
> **Note on Kanban references:** entries below reference Kanban ids from the `agent-teams` project (the dogfood project where this methodology was first validated). When a future dev project under this methodology cites these entries, the referenced incident lives in `agent-teams/shared/decisions.md` — not their own project's log.

## 2026-05-09 — Split `smoke-checklist` / `release-checklist` into methodology (cross-project) + matrix (per-project) — Bucket 4 introduced
**Scope:** bucket-architecture / shared / .claude / context architecture
**Proposed by:** user (during MD compaction Phase 2 — observation: "shared/ มันเป็นของ agent type มากกว่า project")
**Decision:** Introduce **Bucket 4** = `context/leads/<lead>/` — Lead-writable, cross-project per-lead methodology. Sits alongside existing buckets: 1 (DB), 2 (`context/standards/` humans-only), 3 (per-project shared, now formally Bucket 4 in CLAUDE.md numbering). Created `context/leads/dev/smoke-methodology.md` (Tier-1 probe shape, decision matrix, POSITIVE+NEGATIVE rule, restoration discipline, output convention, Kanban #76 worked example, web smoke matrix) and `context/leads/dev/release-methodology.md` (Tier-2 trigger conditions, pre-flight, Step 1-5 patterns, severity scales, wrap-up summary template). Renamed agent-teams's per-project files: `shared/smoke-checklist.md` → `smoke-matrix.md` (now ~50 lines: endpoints/hosts, project-specific trigger paths, canonical seed values, past-escape cross-references) and `shared/release-checklist.md` → `release-matrix.md` (now ~60 lines: endpoint matrix table, Step 4 dep-audit context, Step 5 audit-log queries with agent-teams DB names). CLAUDE.md updated: golden rule extended (Lead writes `context/leads/<lead>/*` too; subagents propose); storage architecture table grew to 4 buckets. `.claude/leads/dev.md` step 5b + Release-wrap-up Step 2 references updated to point at methodology + matrix split. `.claude/agents/dev-tester.md` + `dev-reviewer.md` updated to read methodology FIRST then matrix. `.claude/docs/context-layout.md` updated tree + write/read matrix.
**Reasoning:** Methodology is universal across dev projects (probe shape, severity scales, anti-patterns) but currently lived in `agent-teams/shared/` only. New dev projects scaffolded via POST /api/projects got 3 template files (decisions/api-contracts/db-schema) but NOT smoke-checklist/release-checklist — methodology was inaccessible to them. Lifting methodology to a cross-project per-lead zone solves three problems at once: (a) **runtime efficiency** — Lead loads methodology lazily (only when task triggers smoke/release), saving ~150 lines of context on the ~80% of sessions that don't need it; (b) **edit safety** — single-concern files vs ~250-line mixed lifecycle+methodology playbook reduces accidental cross-concern edits; (c) **lead scaling** — adding novel/data-science/etc. leads tomorrow doesn't bloat their playbook with their own methodology files. User explicitly weighted "accumulate effort during operations > upfront effort" when choosing this architecture.
**Implications:** dev-tester / dev-reviewer spawn prompts now inject TWO files (methodology + matrix). Future dev projects scaffolded via POST /api/projects will read methodology by reference (no copy needed); a follow-up may add per-lead scaffold templates so new projects get a stub `smoke-matrix.md` / `release-matrix.md` with their own endpoints — currently they would need to author these manually (acceptable; matrix is small ~50 lines). Historical decisions.md entries (#78 #80 below) reference `smoke-checklist.md` / `release-checklist.md` by old name — those are factually correct as of their dates and intentionally left unchanged (append-only log). The `backfill-2026-05-08.md` event report refs were patched inline since it's a closed nav-document, not an event log. **Open questions:** (1) should novel projects get a `context/leads/novel/` folder pre-seeded? deferred until first novel project lands; (2) should `api/src/services/project_scaffold.py` dispatch per-lead matrix stubs on POST /api/projects? flagged but out-of-scope for this commit.

## 2026-05-08 — Release wrap-up Tier-2 gate + dev-reviewer security mode — Kanban #80
**Scope:** lead-playbook / agent-prompts / shared
**Proposed by:** lead (gating decision: 'use existing skills' — extend dev-reviewer instead of creating dev-security agent)
**Decision:** Release-wrap-up flow is now an explicit Tier-2 gate triggered by a Kanban task titled `release wrap-up <version>` or `publish wrap-up <version>` (or manual user signal in chat). `.claude/leads/dev.md` got a new top-level `## Release wrap-up flow (Tier-2 gate before publish)` section between the Lifecycle and Anti-patterns sections, codifying 7 sequential steps: (1) pre-flight queue check (no `process_status=2/4` rows allowed), (2) full Tier-1 smoke matrix (all endpoints — extends per-task Tier-1 from #78), (3) `/security-review` slash command (USER-TRIGGERED — Lead documents the request, user fires it manually), (4) dev-reviewer in `mode: security`, (5) `pip-audit` dependency CVE audit, (6) audit-log review on `tasks_history` + projects `updated_at` differential, (7) wrap-up summary `PATCH`ed back into the Kanban task description with GREEN/YELLOW/RED per step. Anti-pattern: running wrap-up in same session as last feature commit → context pollution; open fresh session. New shared template `release-checklist.md` (Lead-only writer) carries the endpoint matrix (POSITIVE+NEGATIVE probe pairs per endpoint), the security-mode audit surface (input validation / authn-authz / SQLi / CSRF / secret leakage / dep CVE / error-disclosure), the SECURITY-* severity scale (BLOCKER/WARN/NIT/KNOWN-GAP — distinct from default review BLOCKER/major/minor/nit), the `pip-audit` severity gate, the audit-log SQL templates, and the wrap-up summary template. `.claude/agents/dev-reviewer.md` got a new `## Security mode (release wrap-up only)` section codifying audit surface + severity scale + `security-mode-review-<date>.md` output path + refuse-on-non-release-task anti-pattern. *Note 2026-05-09:* `release-checklist.md` was later split into `release-methodology.md` (this folder) + per-project `release-matrix.md` — see 2026-05-09 entry above.
**Reasoning:** v0.x has no security gate before publishing. Tier-1 smoke (#78) catches per-task regressions; Tier-2 is the explicit, expensive, rare gate that runs ONLY when releasing. Decision per 2026-05-08 chat: 'use existing skills' route — leverage the built-in `/security-review` slash command (free, user-triggered) plus extend dev-reviewer with a security-mode clause, rather than creating a dedicated `dev-security` agent. Defer dev-security agent until codebase has auth surface that justifies a dedicated role (currently NONE — auth is Phase 4). dev-reviewer's prompt was already only ~100 LOC; adding security mode keeps the file readable and concentrates the security knowledge in one place. The SECURITY-* severity scale is intentionally distinct from default review tags so wrap-up history is filterable across the dev-reviewer/ folder by file-name pattern (`security-mode-review-*.md` vs `review-*.md`).
**Implications:** Future releases follow the release-methodology + project-specific release-matrix. dev-reviewer can be spawned in two modes — default (per-task correctness/style) and security (release-only); the latter refuses on non-release tasks. `pip-audit` is not yet installed in the api container — first wrap-up will install it as a dev dep (one-time setup; will propose adding to `requirements-dev.txt` or pyproject.toml). Authn/authz is documented as SECURITY-KNOWN-GAP for v0.x — first Tier-2 wrap-up will tag it explicitly so the release ships with documented gap rather than accidental omission. Pure process / docs / agent-prompt change — no production code, no schema, no pytest delta. Pairs with #78 (Tier-1) + #79 (regression discipline): #78 catches per-task at deploy-verify; #79 catches at PR-fix; #80 catches at release-wrap-up. Discipline-rollout chain remaining is now #81 (backfill sweep on current branch — see agent-teams's project decisions.md).

## 2026-05-08 — Regression-test discipline (fail-before/pass-after rule) — Kanban #79
**Scope:** agent-prompts / standards proposal
**Proposed by:** lead (after #76 manual application proved the discipline)
**Decision:** Encode the fail-before/pass-after regression-test rule into agent prompts. `.claude/agents/dev-tester.md` got a new `### 2a. Regression test discipline (BLOCKER / MAJOR fixes)` section requiring: (a) the regression test demonstrably FAILS on pre-fix code AND PASSES on post-fix code, (b) both pytest transcripts captured verbatim in a `## Regression demo` section of the final report, (c) `git stash push -- <fix-paths>` workflow to round-trip the production fix while keeping tests in working tree, (d) `# Regression: Kanban #<n>` pin comment (Kanban id only — no commit SHAs that rot), (e) anti-pattern note forbidding bare `actual == baseline` against immutable baselines. The discipline applies only to BLOCKER/MAJOR fixes; feature tests keep the existing golden-path/edges/errors/boundaries workflow. `.claude/agents/dev-reviewer.md` got a parallel audit clause under `### 2. Review` — missing `## Regression demo` section on a BLOCKER/MAJOR task = BLOCKER, vacuous-shape assertion = WARN.
**Reasoning:** Kanban #76 demonstrated that regression tests can pass for the wrong reason — the M9 test asserted `updated_at_after == updated_at_before` after a re-DELETE, but neither side bumped, so equality held trivially. The fix at the test-discipline layer is to mandate the fail-before transcript: dev-tester `git stash`es the production fix and re-runs pytest, capturing the FAIL output before popping. The `git stash` mechanic naturally exercises the "the test must actually probe the bug" invariant — if stashing the fix doesn't break the test, the test isn't load-bearing. Manual application to #76 (in the same session that produced this discipline) caught two strengthened tests failing at exactly the load-bearing assertions on pre-fix code (`tests/test_routes_smoke.py:704` + `:761`); both passed after stash-pop. The evidence base is documented in agent-teams/shared/decisions.md 2026-05-08 #76 entry and serves as the worked example future readers cite.
**Implications:** Every future BLOCKER/MAJOR fix lands with a `## Regression demo` section in dev-tester's report. dev-reviewer enforces. Cost is near-zero at runtime (regression tests live in the existing pytest suite); cost only at fix time is one `git stash` round-trip per fix (~10s). Pairs with Kanban #78 (Tier-1 smoke at deploy-verify): #78 catches at deploy, #79 catches at PR-fix time. Pure agent-prompt change — no production code, no schema, no pytest delta. Standards proposal (humans-only writers) for `context/standards/general.md` Testing section: codify the fail-before/pass-after rule at the project-wide convention level — surfaced here, not auto-edited.

## 2026-05-08 — Tier-1 smoke probe step in dev lifecycle + smoke-checklist template (Kanban #78)
**Scope:** lead-playbook / agent-prompts / shared
**Proposed by:** lead (after #76 escape diagnosis)
**Decision:** Insert lifecycle step 5b in `.claude/leads/dev.md` between current steps 5 and 6: when a task touches `api/src/routers/`, `api/alembic/versions/`, `api/src/schemas/`, `api/src/models/`, `api/src/templates/`, `docker-compose.yml`, env files, or `api/src/main.py`, Lead spawns dev-tester to run scoped `curl localhost:8456` probes against the running container with **behavior assertions** (not just HTTP status codes). New shared template `smoke-checklist.md` (Lead-only writer) holds the decision matrix (which paths trigger Tier-1 vs skip), the probe boilerplate (POSITIVE + NEGATIVE pair shape), the anti-pattern note (vacuous `actual == baseline` shape forbidden — must pair with sibling positive-path assertion), restoration discipline (production rows restored; throwaway rows soft-deleted before return), and the worked example from Kanban #76 (probes A-E). Two agent prompts updated to enforce: `dev-tester.md` got a new `### 2b. Tier-1 smoke probe (live API)` subsection codifying the workflow + the mandatory `## Tier-1 smoke probe results` report section; `dev-reviewer.md` got a Tier-1 audit clause under `### 2. Review` — missing section on a router-touching task = BLOCKER, vacuous-shape assertion = WARN. Cost target: 1-3 probes per task, < 30 seconds total wall-clock. Tier-2 (full API + security + dep audit) stays out of scope — that is Kanban #80. *Note 2026-05-09:* `smoke-checklist.md` was later split into `smoke-methodology.md` (this folder) + per-project `smoke-matrix.md` — see 2026-05-09 entry above.
**Reasoning:** Kanban #76 (projects.updated_at) escaped because (a) the M9 regression test passed for the wrong reason — `actual == baseline` was vacuously true since neither side bumped — and (b) no lifecycle step probed the live API end-to-end after a change landed in the working tree. dev-tester wrote pytest assertions; dev-reviewer audited source; nobody curl'd localhost. Adding step 5b is the minimum process change that catches M9-class vacuous-assertion bugs at deploy-verify without waiting for the regression-test-discipline rollout (Kanban #79). The cost is intentionally low so the step is run every applicable task; richer probes belong in Tier-2 release wrap-up.
**Implications:** Every future task touching the trigger paths above must include a `## Tier-1 smoke probe results` section in dev-tester's report with at least one POSITIVE + one NEGATIVE behavior assertion. dev-reviewer audits for the section. The discipline is now self-documenting via the worked-example pointer to Kanban #76 — future readers can see the canonical probe shape (Probes A-E by dev-backend + Lead's independent no-op probe) without reconstructing it from incident notes. Pure process / docs / agent-prompt change — no production code, no schema change, no pytest delta. Pairs with Kanban #79 (regression-test fail-before/pass-after discipline): #78 catches at deploy-verify; #79 catches at PR-fix time. Tier-2 (Kanban #80) builds on #78's conventions as the release-wrap-up superset. The discipline-rollout chain remaining is now #79 → #80 → #81.
