# PostgreSQL — Audit trail conventions

**Scope:** how to add a per-table append-only audit trail in PG. Applies whenever a business table needs a "who changed what, when" history (UI activity log, debugging, recovery from bad writes). Reflects the pattern shipped in `api/alembic/versions/2026_05_04_2130_initial_schema.py` (`tasks_audit_fn` + `tasks_audit_trg`).

## Table shape

- **Name:** `<entity>_history` (singular history table per audited entity).
- **Columns (minimum):**
  - `id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY`
  - `<entity>_id BIGINT NOT NULL` — **intentionally NOT a foreign key**. History rows must outlive the parent row, so referential integrity here would defeat the purpose. (See the column comment in `api/src/models/task.py` for `tasks_history.task_id`.)
  - `operation CHAR(1) NOT NULL CHECK (operation IN ('U', 'D'))` — extend the CHECK if you add a new op (e.g. `'I'` for an insert audit), never repurpose existing letters.
  - `changed_at TIMESTAMPTZ NOT NULL DEFAULT now()`
  - `snapshot JSONB NOT NULL` — populated as `to_jsonb(OLD)`.
- **Indexes:** at minimum `ix_<entity>_history_<entity>_id` (lookup by parent id) and `ix_<entity>_history_changed_at` (recency / time-range queries).

`<entity>_history` is exempt from the project-wide soft-delete `status` column — audit is append-only by design.

## Trigger pattern

- **Timing:** `AFTER UPDATE OR DELETE`. Never on `INSERT` — the current row already lives in the source table; capturing inserts here would double the write volume for no information gain.
- **Granularity:** `FOR EACH ROW`. Statement-level loses the per-row snapshot.
- **Snapshot:** always `to_jsonb(OLD)`. JSONB makes the audit schema-evolution-proof: when the source table gains/drops/renames columns, old history rows keep their original shape and new rows automatically pick up the new shape — no audit-table migration required.
- **Language:** PL/pgSQL.
- **Naming convention (locked):** function `<entity>_audit_fn`, trigger `<entity>_audit_trg`. e.g., `tasks_audit_fn` / `tasks_audit_trg`. Apply uniformly to every audited table — no per-table variation.
- **Co-location:** define the function and trigger in the same Alembic migration that creates the source table. The audit invariant must be applied atomically with the table — never as a follow-up migration. See `api/alembic/versions/2026_05_04_2130_initial_schema.py` for the canonical block.

## Application code rules

- App code **MUST NOT** `INSERT INTO <entity>_history` directly — let the trigger do it. This is what guarantees out-of-band edits (manual psql, future admin scripts, ad-hoc fixes) end up audited too. A bypass would re-introduce the application-layer bug class the trigger exists to prevent.
- App code **MUST NOT** `UPDATE` or `DELETE` history rows. There is no "edit history" workflow.
- Reads are fine and should go through a dedicated endpoint (e.g. `GET /api/<entity>/{id}/history`) so the audit format is decoupled from internal ORM representation.

## Operation codes — interaction with soft delete

Under the project's soft-delete policy, business tables flip a `status` flag (1→0) instead of issuing SQL `DELETE`. Practical consequences for history rows:

- A soft delete is an `UPDATE` of `status` — the trigger records it as `operation = 'U'`. The `snapshot` JSON shows `status: 1` (the OLD value); recovery means flipping it back.
- `operation = 'D'` becomes rare and signals a **manual cleanup** by an operator running `DELETE FROM <entity> WHERE ...` in psql (e.g. removing a typo'd row before any real users saw it). It should not be produced by application code paths.

If you ever see a `'D'` row from a code path, treat it as a bug — app code is bypassing the soft-delete contract.
