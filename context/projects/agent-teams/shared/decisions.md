## 2026-05-10 — Kanban #694 Phase 2 closed: drop `ux_projects_active_one` + remove PATCH atomic-clear + `GET /api/projects/active` → 410 Gone
**Scope:** api / db / tests / shared
**Proposed by:** dev-backend (migration 0006 + ORM/router edits + 3 regression tests + 27 test-URL substitutions; pytest 121 → 124 GREEN; round-trip up→down→up verified — seeded agent-teams row id=1 byte-identical pre/post) → dev-tester GREEN+1YELLOW (6/6 Tier-1 probes — multi-active confirmed, 410 wire-body byte-equal, agent-teams id=1 byte-identical pre/post all 5 PATCH ops; YELLOW = OpenAPI 410 not auto-documented, closed in same slice) → dev-reviewer GREEN (0 BLOCKER / 0 MAJOR / 0 MINOR / 1 NIT — stale comments referencing dropped index, closed in same slice)
**Decision:** Three coupled changes land in one slice now that Phases 1 (CLAUDE.md bootstrap rewrite, commit `e0939e0`) and 3 (X-Project-Id header gate, commit `83d9314`) are shipped:
- **Migration `0006_drop_active_one`** drops the partial unique index `ux_projects_active_one ON projects(is_active) WHERE is_active IS TRUE AND status = 1`. The "single active project" invariant the index enforced is obsolete — each Claude Code session now binds to a project by name at bootstrap, so terminals run in parallel against different projects each carrying `is_active=true`. The SQLAlchemy ORM `Index(...)` declaration in `api/src/models/project.py` was removed in the same commit so app and migration stay in sync. Downgrade restores the index with byte-identical predicate (verified against the 0002 migration that originally shipped it).
- **PATCH `/api/projects/{id}` atomic-clear removed.** `_clear_other_active` helper and BOTH call sites (POST `create_project` when `is_active=true` AND PATCH `update_project` when `is_active=true`) gone. The unused `from sqlalchemy import update` import dropped. N7 no-op-skip / `updated_at` bump / IntegrityError-409-translation on `ux_projects_name_active` paths are untouched.
- **GET `/api/projects/active` → 410 Gone** with source-text-locked detail string `"Endpoint deprecated. Use /api/projects/by-name/{name} or /api/projects?status=1 instead."` per #122 pattern. Locked by both an HTTP wire test and a source-text-lock test. The route decorator now carries `responses={410: {"description": ...}}` so the deprecation is publicly discoverable in `/openapi.json` — runtime `raise HTTPException(...)` does NOT auto-document, the explicit decorator kwarg is mandatory for clients consuming the OpenAPI schema (typed clients, contract tests, API explorers).
- **DELETE `/api/projects/{id}` side effect retained, rationale refreshed.** The `is_active=true → false` clear inside `delete_project` survives — not because of any unique-index constraint (gone), but because a soft-deleted row should not advertise itself as active in any list/by-name query. Inline comments in `delete_project` (docstring + L268) refreshed to ground the rationale in read-side consistency rather than the dropped index.

**Reasoning:** The session-scoped active project model (Kanban #694 Phase 1 / #695 Phase 3) makes multi-active-row legitimate — each terminal binds to its own project. Keeping the partial unique index would silently fail PATCH `is_active=true` on a second project once any first project was active. Keeping the atomic-clear would silently STOMP the first session's active flag the moment a second terminal bound to a different project. Both behaviors were load-bearing on the old "single active" invariant and had to go together; doing it as one slice avoids half-states. The 410 (over a permanent redirect or 404) keeps the deprecation visible — Lead bootstrap MUST update; silent fallback would mask the migration of every existing client. The DELETE-side `is_active=false` clear is preserved because read-side queries (list / by-name) treat `is_active=true` as a "this project is live" signal — a soft-deleted row claiming to be live is incoherent regardless of whether any unique constraint enforces it.

**Implications:**
- Multiple rows may carry `is_active=true` simultaneously. Frontend (Phase 3 / V2 #406) consumes `/api/projects?status=1` to render the live-projects list; bootstrap clients use `/api/projects/by-name/{name}`.
- Test suite: pytest 121 → 124 (+1 wire-410 test, +1 source-text-lock test, +1 multi-active regression test pinned `# Regression: Kanban #694`; 27 URL substitutions inside existing tests from `/api/projects/active` to `/api/projects/by-name/agent-teams` since most usages were just fetching the seeded project's id).

- Standards-propagation queue grows by THREE (humans-only writers): (a) `context/standards/fastapi/atomic-mutations.md` cites the just-removed `_clear_other_active` pattern as its canonical worked example — HISTORICAL now; reframe ("agent-teams used this pre-694; the txn-atomicity principle still applies to other singleton-flag use cases") or replace with a different exemplar; (b) `context/standards/sqlalchemy/migrations.md` cites `ux_projects_active_one` as the partial-unique-migration worked example — same fate; the natural replacement is `ux_projects_name_active` (still live, same partial-unique form); (c) NEW codify rule at `context/standards/fastapi/routing.md` Deprecation section: "when an endpoint raises a non-2xx status as its only response, declare it via `responses={code: {description: ...}}` on the route decorator — `/openapi.json` does NOT auto-discover runtime `raise HTTPException(...)` codes" (lesson surfaced from dev-tester YELLOW; would have shipped silently as a `/openapi.json` 200-only spec without it).

## 2026-05-10 — `?pending=true` shortcut on GET /api/tasks — Kanban #697 closed
**Scope:** api / tests / shared
**Proposed by:** dev-backend (15-line router edit + 5 regression tests, 358 lines added) → dev-tester GREEN (6/6 Tier-1 probes — POSITIVE+NEGATIVE pair plus OpenAPI contract verification; throwaway project 560 + tasks 698-702 soft-deleted; agent-teams id=1 byte-identical pre/post) → dev-reviewer GREEN (0 BLOCKER / 0 MAJOR / 0 MINOR / 2 NIT advisory)
**Decision:** `GET /api/tasks` gains a `pending: bool = Query(default=False)` shortcut. When `pending=true`, the router filters `WHERE process_status != TaskStatus.DONE` (i.e., IN (1,2,3,4)). When BOTH `pending=true` AND `process_status=N` are provided, the explicit `process_status` wins (more specific) and `pending` is silently ignored. Precedence is structurally enforced by control flow — `if process_status is not None: ... elif pending: ...` — mirroring the existing `if top_level_only: ... elif parent_task_id is not None: ...` pattern two clauses below. The clause uses the named constant `Task.process_status != TaskStatus.DONE`, NOT the bare literal `5`. **Out of scope (deliberate):** multi-value `process_status` (kept as int), name-based filter (todo/in_progress/...), any change to existing `process_status` semantics. Frontend (#406) consumes this later — no FE impact today.
**Reasoning:** Lead's bootstrap workflow needs "list pending tasks" every turn — current API only accepts a single int `process_status`, and FastAPI silently ignores unknown query params. A guess like `?status=todo,in_progress,...` returned 200 + ALL rows including done — false-positive that wasted ~3 prompts per session on a Python-side fallback to filter through `docker compose exec -T api python -c ...`. Exposing the most common Lead query as a first-class param eliminates the fallback path entirely. The `elif`-precedence (vs boolean arithmetic) is deliberate — a future drift to `if pending:` would silently re-enable the false-positive class; the precedence test (`test_list_tasks_pending_and_process_status_explicit_wins`) seeds BOTH ps=5 AND ps≠5 rows and locks `?pending=true&process_status=5` → exactly 1 row with `process_status==5`, so any `if`/`elif` regression fails loudly.
**Implications:** Lead bootstrap can now use `curl -H "X-Project-Id: <id>" "/api/tasks?pending=true"` directly — no Python fallback. Convention propagated: future convenience-shortcut bool params on list endpoints MUST cede precedence to the more-specific explicit param via control-flow `elif`, not boolean arithmetic. Standards-propagation queue grows by one (humans-only writers): codify the convenience-shortcut + `elif`-precedence pattern at `context/standards/fastapi/routing.md` (List endpoints — Filters section). pytest: 116 → 121 (+5 new regression tests, all pinned `# Regression: Kanban #697`). No migration; no schema change. Cross-references: dev-tester probe transcript in `dev-tester/current-state.md`; dev-reviewer 2 NITs (NIT-1: no source-text-lock test for the SQL clause; NIT-2: tests don't backlink to api-contracts.md) deferred — both advisory, neither gating.
**OpenAPI contract:** `pending` param now publicly discoverable at `paths./api/tasks.get.parameters` with `schema.type:"boolean"`, `default:false`, and a description that documents both the predicate (`process_status != 5`) and the precedence rule. Verified by Tier-1 probe OPENAPI.
**Files:** `api/src/routers/tasks.py` (+15 lines: Query signature 66-75 + `elif` filter clause 92-96), `api/tests/test_routes_smoke.py` (+358 lines: 5 regression tests). api-contracts.md `GET /api/tasks` Query line updated atomically with this entry.

## 2026-05-09 — #483 implementation: tasks.run_mode + grant-consent endpoint + cross-table validator (#481-B)
**Scope:** api / tests / shared
**Proposed by:** dev-backend (ORM + Pydantic + endpoint + validator + 31 tests) → dev-reviewer GREEN (0 BLOCKER / 0 MAJOR / 2 MINOR / 4 NIT — all defer-able) → dev-tester Tier-1 GREEN 8/8 (live API; throwaway project; agent-teams id=1 byte-identical pre/post)
**Decision:** Wired migration 0005's columns through the stack with these implementation choices locked at the project layer (the cross-team-applicable framing lives in `context/teams/dev/decisions.md` 2026-05-09 'Kanban-driven AI: 2-mode model + per-project consent gate'):
- **Constraint name** mirrored: `ck_tasks_run_mode_valid` in both the migration AND the ORM `CheckConstraint` declaration (pattern from `_PROJECT_TEAM_ALL` / `ck_projects_team_valid` lockstep, Phase 2.5b1).
- **`POST /api/projects/{id}/grant-consent`** — body `{"confirm_name": "<name>"}` with Pydantic `extra="forbid"` (NOT default `extra="ignore"`). A typed-acknowledgment endpoint must fail loud on smuggled fields — `extra="ignore"` would defeat the deliberate-action UX premise. 400 on mismatch with source-text-locked detail string `"confirm_name must match project name exactly"`. 404 on missing OR soft-deleted project (parity with `get_or_404 status=ACTIVE`). 422 on extra fields.
- **Idempotent re-grant** locked: read `project.auto_run_consent_at` and short-circuit BEFORE assigning `func.now()` if non-null. The first consent is the auditable timestamp; re-grant must not bump `auto_run_consent_at` OR `updated_at`. Verified at the wire layer by Tier-1 probe E (re-grant after `sleep 1` → both timestamps byte-identical).
- **Cross-table validator location:** `src/services/run_mode.py::assert_consent_for_run_mode(db, project_id, run_mode)` — service-layer helper called from POST + PATCH `/api/tasks`. NOT a DB CHECK because it spans tables. Reads only `Project.auto_run_consent_at` (single-column scalar select with `Project.status == ACTIVE`) — keeps the hot POST/PATCH path cheap.
- **PATCH resolved-final-mode rule:** the validator fires on the RESOLVED final `run_mode` — `payload.run_mode if "run_mode" in updates else task.run_mode`. Downgrading from `auto_headless` to `manual` always succeeds (resolved=manual → no fire). PATCH on an `auto_headless` row when consent is gone fails (resolved=auto_headless re-validates and rejects) — forces the operator to either downgrade run_mode first OR re-grant consent.
- **Lockstep guard:** `TaskRunModeLiteral` ↔ `TaskRunMode.ALL` import-time guard at the bottom of `schemas/task.py`, mirroring the existing `TeamCode` ↔ `ProjectTeam.ALL` guard in `schemas/project.py:115-119`. Uses `RuntimeError` (not `assert`) so it survives `python -O`. Drift test in `tests/test_run_mode_consent.py` monkey-patches `TaskRunMode.ALL` → reloads schemas module → asserts RuntimeError.
- **Source-text-locks:** two new lock tests pin (a) `"confirm_name must match project name exactly"` in `routers/projects.py`, (b) the consent-required template `"project {project_id} has not granted auto-headless consent"` in `services/run_mode.py`. Pattern per #122.
- **Test count:** 31 new tests in `tests/test_run_mode_consent.py`. pytest 71 (baseline) + 31 (#483) + 2 (#482 isolation pinning) = **102 passed** against fresh `agent_teams_test` (per Issue 2). The previously-flaky `test_list_tasks_default_filters_active_only` now passes deterministically.
- **`MINOR-1` follow-up filed** (Kanban TBD): when `POST /api/tasks` carries `run_mode='auto_headless'` AND `project_id` references a missing or soft-deleted project, the consent error masks the FK error (same operator mistake produces two different detail strings depending on `run_mode`). Wire-contract drift, not a bug. Acceptable to ship #483 as-is and disambiguate in a follow-up SELECT branch.
- **`MINOR-2` follow-up:** `tests/test_run_mode_consent.py:54-65` `_reset_consent` helper uses direct ORM `update()` to simulate consent revocation (no public revoke-consent endpoint yet). Justified deferral; remove when `POST /api/projects/{id}/revoke-consent` ships.

**Reasoning:** All five items above came up as live design questions during implementation. The idempotent-re-grant rule was specced in the team-methodology decision but the implementation needed a real short-circuit before `func.now()` — without it, re-grant would re-stamp despite the "lean toward 200-with-existing-row" wording. The PATCH resolved-mode rule was the trickiest — initial drafts asserted only on `payload.run_mode` (which would let PATCH-other-fields-on-headless-task slip past); re-reading task #483's spec made it clear the validator must consider the FINAL post-PATCH state. The 404-on-soft-deleted decision is consistent with the existing `get_or_404 status=ACTIVE` pattern on `/api/projects/by-name/{name}`; a soft-deleted project is by definition not in active use and shouldn't grant consent.

**Implications:**

- Frontend (#484) gets stable types: `run_mode: "manual"|"auto_pickup"|"auto_headless"` (Literal, default `"manual"`), `auto_run_consent_at: string | null` (ISO-8601). Grant-consent body `{confirm_name: string}` with `extra="forbid"` enforced at the API.

## 2026-05-09 — Test-database isolation (`agent_teams_test`) — Issue 2 of incident response
**Scope:** api / tests / dev tooling
**Proposed by:** dev-backend (after Lead diagnosed leak source — pytest writing to live DB) → dev-reviewer GREEN (0 BLOCKER / 0 MAJOR / 0 MINOR / 2 NIT advisory)
**Decision:** Tests run against a per-pytest-session ephemeral database named `agent_teams_test`. Lifecycle: (1) `tests/conftest.py` sets `DATABASE_URL` to `…/agent_teams_test` at module top — BEFORE any `from src import …` (because `src.db.engine` is a module-level singleton built from `get_settings().database_url` at import time; the override must fire first). (2) Session-scoped `autouse` fixture `_setup_test_database` connects to the maintenance `postgres` DB, runs defensive `pg_terminate_backend` on any leftover `agent_teams_test` connections, drops + creates `agent_teams_test`, runs `alembic upgrade head` (subprocess so the sync alembic API stays out of the async event loop), runs the existing `scripts.seed._seed()`, disposes the engine for event-loop hygiene, then yields. (3) Teardown drops the test DB so the next pytest invocation starts clean. (4) Tests MAY leave data within the test DB during the session (no transaction-rollback wrapper). All four pre-existing fixtures (`_reset_engine_pool_per_test`, `client`, `scaffold_cleanup`, `db_session`) preserved unmodified. Two contract tests added at `tests/test_db_isolation.py` pinning the invariant — `engine.url` must contain `agent_teams_test`, AND a round-trip via SessionLocal must report `current_database() = 'agent_teams_test'`. If a future refactor moves the env override below a `from src import …` line, both pinning tests fail loudly.
**Reasoning:** Audit on 2026-05-09 found the live `agent_teams` DB had grown from "100s" of tasks to **636 tasks (32 active + 604 soft-deleted) + 510 projects (39 active + 471 soft-deleted)** — pytest had been writing every run for ~2 days, soft-deleting but never reaping. Test names like `qa-smoke-lifecycle (test row, safe to delete)` × 53 confirmed 53 invocations had each leaked a row. End-to-end real-system verification (audit triggers fire, soft-delete partial-unique semantics exercised, FK cascade behavior covered) outweighs the cost of intra-session test data leftover; the user explicitly accepted that trade-off ("มี test data ได้เลยไม่ติดปัญหา จะได้รู้ว่ามันทำงานเข้าระบบได้ถูกต้องจริงๆ ด้วย"). Per-test transaction rollback rejected — audit triggers fire-then-roll-back, which would silently break any test asserting on `tasks_history` row counts. Truncate-per-session rejected — doesn't isolate parallel pytest invocations from each other. Drop+create per session is per-invocation isolated by construction.
**Implications:** pytest no longer touches the live DB — load-bearing assertion verified by dev-backend (live count `636/510` identical before AND after pytest, idempotent across two consecutive runs). The previously-flaky `test_list_tasks_default_filters_active_only` now passes deterministically (was failing because dev DB had >500 task rows pushing the soft-deleted row past `limit=500`; fresh test DB has only seed rows). Issue 3 (cleanup of the 604 + 471 soft-deleted live rows from prior runs) becomes one-time work — runs no longer pollute. Issue 3 is a **user action** per the raw-SQL-DML hard rule (Issue 1, commit `59f824c`): subagents propose the cleanup statement, user runs it. Standards-propagation queue grows by two: (a) `python/testing.md` "Pre-import env overrides for module-level engines" pattern; (b) `postgresql/test-database.md` "drop+create per pytest session preferred over tx-rollback when audit triggers / partial unique / FK cascade are part of the contract under test." Both flagged for humans-only writers. Hook (`.claude/hooks/block-raw-sql-dml.ps1`) does NOT fire on the fixture's CREATE/DROP DATABASE statements — those go through async SQLAlchemy `text()`, not Bash `psql -c` (different tool boundary).

## 2026-05-04 — BigInteger autoincrement, not UUID, for primary keys
**Scope:** backend / db / shared
**Proposed by:** backend
**Decision:** All tables use `id BIGINT GENERATED BY DEFAULT AS IDENTITY` (SQLAlchemy `BigInteger autoincrement=True`) instead of `uuid DEFAULT gen_random_uuid()`.
