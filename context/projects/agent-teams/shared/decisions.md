**Decision:** Adds `tasks.parent_task_id BIGINT NULL` self-referential FK with full app-layer validation. Migration `0003_tasks_parent_task_id` (applied 2026-05-08 16:00): `add_column` + `ON DELETE CASCADE` FK `fk_tasks_parent_task_id` + CHECK `ck_tasks_parent_task_id_not_self` (`parent_task_id IS NULL OR parent_task_id <> id`) + index `ix_tasks_parent_task_id`. ORM uses canonical adjacency-list pattern with string `remote_side="Task.id"` (survives circular-ref import order) + `parent` / `subtasks` relationships, both `lazy='select'`. Pydantic: `TaskCreate.parent_task_id: int | None = Field(default=None, ge=1)`; `TaskRead.parent_task_id` exposed; `TaskUpdate.parent_task_id` REJECTED via `@model_validator(mode='after')` checking `if "parent_task_id" in self.model_fields_set` — explicit-null and explicit-int both 422 (locked by paired test cases per #79 discipline + the W1 null-case extension). Router: POST validates parent existence + `parent.status=ACTIVE` + `parent.project_id == payload.project_id` (400 with stable details `parent_task_id <n> does not exist or is deleted` and `parent_task_id <n> belongs to a different project` — pinned via M5/#122 paired-tuple source-text-lock test); DELETE blocks 409 with stable detail `Cannot delete task — <n> active subtask(s) reference this task` if any active children reference the row, AFTER the idempotent re-DELETE early-return so already-deleted parents still 204; GET adds `?parent_task_id=N` (filter to direct children) and `?top_level_only=true` (filter to `parent_task_id IS NULL`) — when both provided, `top_level_only` takes precedence and `parent_task_id` is silently ignored (locked in docstring + api-contracts.md).
**Reasoning:** **(a) Soft-delete parent with active children → 409 (block, not cascade-soft-delete)** — cascade-soft-delete on a 50-child umbrella is too easy to invoke by accident; 409 forces the operator to decide (delete each child first, or keep the parent live). **(b) Same-project enforced at app layer** — composite FK across (project_id, id) would be DB-cleaner but adds two-column FK complexity for a constraint we can express in 3 lines of Python at the only entry point that creates rows. **(c) Re-parenting NOT allowed in V1** — re-parenting introduces ordering / cycle questions that aren't worth solving until Phase 3 UI demands them. The `model_validator` REJECT-BY-PRESENCE pattern (vs `extra='ignore'` silent-drop) is required so silent client-side bugs surface as 422 instead of as silently-dropped fields. **(d) FK `ON DELETE CASCADE`** — app never hard-deletes (only soft-delete via `status=0`), so CASCADE never fires from the app path. It's a defense-in-depth backstop for raw-SQL drift. **(e) No status rollup** — UX may compute a derived "umbrella status" on display; baking it into the DB couples write paths to UX policy that's still TBD.
**Implications:** Phase 3 UI (Kanban #3) can now create true parent/child task relationships — V2 (board) and V3 (project switcher) are fileable as subtasks of #3. Tier-1 smoke matrix gained 8 probes (POSITIVE 5 + NEGATIVE 3) covering the new POST/PATCH/DELETE/GET behaviors — see Tier-1 transcript in `dev-tester/current-state.md`. Standards-propagation queue grew by THREE: (a) codify the Pydantic `model_validator` REJECT-BY-PRESENCE pattern at `pydantic/v2-conventions.md` Settings/Update section — `extra='ignore'` silent-drops + `if x is not None` misses explicit-null + only `model_fields_set` correctly differentiates "not provided" from "provided as anything"; (b) codify the SQLAlchemy adjacency-list pattern with string-form `remote_side` at `sqlalchemy/orm.md` self-ref section; (c) extend `general.md` Testing section — Update-schema-REJECT pattern tests MUST cover BOTH `{field: value}` AND `{field: null}` cases (single-case is the Kanban #76 vacuous-assertion class). ## 2026-05-08 — projects.name path-traversal hardening — Kanban #121 closed
**Scope:** api / tests / shared
**Proposed by:** dev-backend (two-layer fix) → dev-tester (regression locks + #79 fail-before/pass-after demo + #78 Tier-1 smoke) → dev-reviewer (audit GREEN: 0 BLOCKER / 0 WARN / 0 NIT)
**Decision:** Closes SECURITY-WARN W1 from backfill #81. Both layer-2 guards `return False` (never raise) per existing scaffold contract. No new imports. Tests strengthened: 3 new regression tests in `tests/test_routes_smoke.py` (`# Regression: Kanban #121` pins at L951/L996/L1026): `test_post_project_rejects_path_traversal_names` (6-case charset matrix), `test_patch_project_rejects_path_traversal_names` (PATCH lockstep), `test_scaffold_service_rejects_traversal_directly` (uuid-suffixed paths + pre-clean to defeat leaked-dir contamination from prior fail-before runs; locks `result is False` AND `not dangerous_path.exists()`). **Reasoning:** Two-layer chosen over single-layer because each layer covers a distinct threat model: (a) HTTP-borne attack via FastAPI/Pydantic (caught at layer 1, fast-path 422 with no I/O); (b) future internal caller — admin script, scheduled job, ORM-direct creation path — that bypasses Pydantic (caught at layer 2). Standards-propagation queue grew by two: (a) codify the two-layer path-traversal defense pattern (boundary regex + service-layer `is_relative_to` containment) at `context/standards/python/path-handling.md`; (b) codify the uuid-suffix-per-case + pre-clean idiom for FS-mutating regression tests at `general.md` Testing section — Test C surfaced that deterministic dangerous paths cause leaked-dir-from-prior-fail-before-run to mask pass-after on the next invocation.
